I denna del använde jag git revert för att ångra en tidigare förändring i projektets historik.

Först försökte jag ångra committen "docs: improve serving instructions" med kommandot:

git revert 7d9c413

Detta ledde till en konflikt i filen recept.md, eftersom filen hade ändrats senare i historiken och Git inte kunde ångra committen automatiskt.

Efter det avbröt jag försöket med:

git revert --abort

Sedan valde jag en annan commit som var lättare att ångra:

git revert 1792459

Den committen var:
feat: add garlic variant

Jag valde git revert istället för git reset eftersom revert är säkrare och inte skriver om historiken. Git revert skapar en ny commit som ångrar en tidigare förändring, medan git reset ändrar historiken och kan vara riskabelt om projektet redan är pushat eller delas med andra.

Efter revert skapades en ny commit:
Revert "feat: add garlic variant"

Det betyder att den ursprungliga committen finns kvar i historiken, men att dess effekt har tagits bort genom en ny commit.

Jag valde denna lösning eftersom den gör historiken tydlig, spårbar och säker.


Jag skapade en patch med kommandot:

git format-patch -1 -o patches/

Patchfilen sparades i katalogen patches/.

Sedan tog jag bort committen med:

git reset --hard HEAD~1

Efter det applicerade jag patchen med:

git apply patches/*.patch

Skillnaden mellan git apply och git am är:

- git apply applicerar endast ändringarna i filerna och skapar ingen commit automatiskt
- git am applicerar patchen och skapar en commit automatiskt med commit-meddelandet

Jag valde git apply eftersom jag ville ha kontroll över committen och skriva commit-meddelandet själv.

Detta visar hur man kan exportera och återanvända förändringar i ett projekt.
